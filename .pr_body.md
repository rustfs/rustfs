## Summary of Changes
This PR implements **prepared IAM authorization** and **object-tag-aware policy evaluation** so conditions that depend on existing object tags are evaluated with the correct tag map.

- **`crates/iam`**: Add `prepare_auth` / `eval_prepared` (`PreparedIamAuth`) so merged identity policies and STS/session policies are prepared once, `s3:ExistingObjectTag` usage is detected from policy documents, and evaluation reuses that context. Replace the old STS entry path with `prepare_sts_auth` + `eval_prepared`; service accounts use `prepare_service_account_auth` + `eval_prepared`. `is_allowed` delegates to this pipeline.
- **`crates/policy`**: Add helpers to detect `ExistingObjectTag` / `s3:ExistingObjectTag` condition keys in IAM and bucket policies (including structured checks so arbitrary substrings in values do not spuriously trigger tag loads). Add regression tests for condition-key detection.
- **`rustfs/src/storage/access.rs`**: In `authorize_request`, merge object tags into condition values when IAM preparation or the bucket policy indicates they are needed; cache tags per request `(bucket, object, version_id)`; parse bucket policy JSON when possible to decide whether tag conditions apply (malformed JSON with the marker still conservatively loads tags). Remove per-handler prefetch of tag conditions in favor of this centralized path.
- **`crates/e2e_test`**: Add STS `AssumeRole` and bucket/IAM policy E2E coverage for `ExistingObjectTag`; extend common STS helpers and teardown.
- **Admin**: Document why canned policies with an empty `Version` are rejected (`rustfs/src/admin/handlers/policies.rs`); minor `rustfs/src/admin/auth.rs` doc cleanup. Policy-crate tests support admin permission semantics.
