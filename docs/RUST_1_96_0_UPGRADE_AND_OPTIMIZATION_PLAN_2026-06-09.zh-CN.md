# RustFS Rust 1.96.0 升级与优化实施方案

## 1. 目标

本方案用于将 RustFS 工作区的 Rust 基线从 `1.95.0` 升级到 `1.96.0`，并在尽量低风险的前提下，把此次升级带来的直接收益落到仓库中。

本次处理目标分为三类：

1. 工具链与 CI 对齐：避免本地、CI、文档示例出现版本漂移。
2. 小步代码优化：只引入 Rust `1.96.0` 可直接受益、且不会改变业务语义的改动。
3. 升级说明沉淀：把收益、风险、验证项和回滚思路写入 `docs/`，便于后续 review、PR 和 rollout。

## 2. 当前基线

升级前的工作区基线如下：

1. 工作区 `rust-version` 为 `1.95.0`。
2. `rust-toolchain.toml` 使用 `stable` 浮动通道。
3. GitHub Actions 中多个 job 使用 `stable` 或隐式稳定版。
4. 局部文档示例仍显式写着 `cargo +1.95.0 ...`。

这意味着当前存在两个问题：

1. 版本声明与实际执行版本可能漂移，排查 CI 差异时不够稳定。
2. 即使仓库升级到 `1.96.0`，如果不消费新能力，本次升级的实际收益会偏弱。

## 3. Rust 1.96.0 对 RustFS 的实际收益判断

### 3.1 值得关注的收益

#### 3.1.1 `assert_matches!` / `debug_assert_matches!` 稳定

这是对 RustFS 最直接、最容易落地的收益。

仓库中已存在大量：

```rust
assert!(matches!(value, Pattern));
```

升级后可逐步替换为：

```rust
assert_matches!(value, Pattern);
```

价值在于：

1. 断言失败时输出更直观，测试定位成本更低。
2. 对状态机、错误枚举、并发路径、协议分支测试更友好。
3. 不改变业务逻辑，仅改善测试可观测性，回归风险低。

#### 3.1.2 `core::range::*` 新类型稳定

这项能力对长期 API 设计有帮助，但对现有代码不是立刻自动生效的收益。

适合未来在以下场景逐步采用：

1. 偏移区间、字节窗口、扫描范围的值对象建模。
2. 需要把区间本身在多层函数之间传递而不是只做切片语法糖的代码。

当前判断：

1. 可以纳入后续设计演进方向。
2. 不建议在本次升级分支里大规模推广，避免把 MSRV 升级做成语义重构。

#### 3.1.3 Cargo 供应链安全修复

Rust `1.96.0` 同期带入了 Cargo 针对第三方 registry 处理的安全修复。

对 RustFS 的意义主要是：

1. 未来如果引入私有 registry、镜像源、企业供应链治理，安全基线更稳。
2. 对 release/构建环境的长期维护价值高于眼前功能收益。

### 3.2 不应夸大的收益

以下收益不应作为本次升级的主理由：

1. 核心数据面吞吐量显著提升。
2. async/runtime 层大幅提速。
3. 大量历史代码自动简化。

本次升级更偏向“工程稳定性、测试诊断、工具链一致性”的改进，而非一次性能型升级。

## 4. 本次分支建议落地内容

### 4.1 必做项

1. 将工作区 `rust-version` 提升到 `1.96.0`。
2. 将 `rust-toolchain.toml` 从浮动 `stable` 固定到 `1.96.0`。
3. 将 GitHub Actions 中使用的稳定通道显式固定到 `1.96.0`。
4. 更新文档中显式写死的 `+1.95.0` 命令示例。

### 4.2 推荐的小步优化

本次已选择测试代码作为首批消费点，原因是：

1. 风险最低。
2. 与 1.96.0 的 `assert_matches!` 收益强相关。
3. 不改变运行时语义。

建议优先改造的区域：

1. 错误枚举断言测试。
2. 分布式锁重试/超时/故障类型测试。
3. 表目录、元数据维护、冲突检测类测试。

### 4.3 本次不建议做的事

1. 不把 `Range` 新类型引入到核心业务 API 中做大面积重构。
2. 不借升级顺手做 clippy 大扫除。
3. 不在同一分支里混入依赖升级、架构迁移或大规模格式化噪音。

## 5. 风险分析

### 5.1 工具链锁定风险

把 `stable` 固定为 `1.96.0` 后，优点是可重复性提升，但也意味着：

1. CI 不再自动跟随新稳定版。
2. 后续若要升到 `1.97.x`，需要显式提交版本变更。

这是一个有意为之的取舍，适合 RustFS 当前偏重稳定构建与可复现的仓库形态。

### 5.2 宏导入兼容性风险

`assert_matches!` 需要依赖 `std::assert_matches::assert_matches`。

风险点主要在于：

1. 如果局部模块有命名冲突，需要显式调整导入。
2. 若未来需要兼容更老工具链，会阻断编译。

由于本次就是将 MSRV 提升到 `1.96.0`，该风险是可接受的。

### 5.3 CI 漂移风险

如果只改 `Cargo.toml` 而不改 workflow，会出现：

1. 本地按 `1.96.0` 编译通过。
2. CI 仍跑 `stable`，未来随稳定版前进而产生不一致。

因此本次必须同步升级 workflow 入口。

## 6. 验证矩阵

### 6.1 最低验证

1. `cargo fmt --all`
2. `cargo fmt --all --check`
3. `cargo test -p rustfs-lock`
4. `cargo test -p rustfs --lib table_catalog`

### 6.2 仓库要求验证

按仓库根 `AGENTS.md` 要求，代码改动合入前应跑：

```bash
make pre-commit
```

若本地时间或环境受限，至少应明确说明未完成项，并避免在验证失败状态下推进 PR。

### 6.3 CI 对齐验证

需要特别关注以下链路：

1. `.github/actions/setup` 是否正确安装 `1.96.0`。
2. `ci.yml` 中显式 `dtolnay/rust-toolchain` 步骤是否全部对齐。
3. `nix.yml` 中 `cargo` user-agent 版本号是否同步更新。

## 7. 回滚与降级思路

若升级后出现广泛兼容性问题，可按以下顺序回滚：

1. 回退测试侧 `assert_matches!` 使用。
2. 回退 CI/toolchain 固定版本。
3. 回退工作区 `rust-version`。

建议不要只回退局部其中一项，否则容易形成“声明版本”和“执行版本”不一致的中间状态。

## 8. 推荐 rollout 顺序

1. 先在独立分支完成版本与测试改造。
2. 运行最小验证，确认本地通过。
3. 跑 `make pre-commit`。
4. 提交 PR，让 CI 完整验证。
5. 若 CI 稳定，再继续扩展第二批 `assert_matches!` 改造。

## 9. 本次分支的建议结论

本次 Rust `1.96.0` 升级是值得做的，但应定位为：

1. 一次工程基线升级。
2. 一次测试诊断能力增强。
3. 一次工具链一致性修复。

不应把它包装成一次显著的数据面性能升级。

如果后续要继续放大收益，建议分下一阶段单独处理：

1. 第二批测试断言迁移。
2. `core::range::*` 在局部 API 中的设计性试点。
3. 结合 CI/benchmark 结果再评估是否存在编译性能或 lint 体验层面的附加收益。
