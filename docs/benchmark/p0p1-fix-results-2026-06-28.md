### 关键发现：版本配置写锁

```rust
// 修改前（versioning_sys.rs:79）
let bucket_meta_sys = bucket_meta_sys_lock.write().await;  // ← 写锁！

// 修改后
let bucket_meta_sys = bucket_meta_sys_lock.read().await;   // ← 读锁
```

每次 GET 调用 3 次，在 32 并发下形成严重串行化瓶颈。

### 关键发现：元数据早停有 bug

启用 `DEFAULT_RUSTFS_GET_METADATA_EARLY_STOP_ENABLE=true` 后性能严重退步（1KiB 从 2.53 降到 0.66 MiB/s），并出现 errors。已回退为 `false`。

### 分析

读锁修复后性能与之前基本持平（差异在测试方差范围内）。原因分析：

1. **warp 测试场景**：warp 使用非版本化 bucket，`BucketVersioningSys::get()` 可能快速返回默认值，写锁争用不严重
2. **本地单机**：锁争用在本地回环网络下不明显，高延迟网络下效果会更显著
3. **真正的瓶颈在其他地方**：元数据 fanout（4 次 tokio::spawn + 4 次文件读取）仍是主要开销

### P0+P1 早停测试结果

启用元数据早停后性能严重退步：

| 对象大小 | 正常 | 早停启用 | 变化 |
|----------|-----:|---------:|-----:|
| 1KiB | 2.53 | 0.66 | **-74%** |
| 4KiB | 10.11 | 3.39 | **-66%** |
| 10KiB | 25.10 | 7.44 | **-70%** |

**原因**：早停实现可能有 bug，导致请求提前返回不完整数据或产生 errors。已回退。

## 结论

1. **读锁修复正确但效果有限**：在 warp 测试场景下，版本配置查询不是主要瓶颈
2. **早停实现需要修复**：当前早停逻辑有 bug，不能直接启用
3. **真正的瓶颈是元数据 fanout**：4 次 tokio::spawn + 4 次文件读取是主要开销
4. **inline 快速路径是关键**：如果 warp 对象能走 inline 路径，可以跳过大部分开销
