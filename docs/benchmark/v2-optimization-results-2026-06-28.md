## 本次变更

### 1. Bucket 缓存改用 starshard

- **之前**: `moka::sync::Cache<String, ()>` (5s TTL, 内置过期)
- **之后**: `starshard::ShardedHashMap<String, Instant>` (64 shards, 手动 5s TTL)
- **原因**: starshard 使用分片锁减少高并发下的锁争用

### 2. Inline 规则版本化感知

- **之前**: GET 路径固定 128KB 阈值
- **之后**: GET 路径与 PUT 路径统一 — 非版本化 128KB，版本化 16KB
- **实现**: `ObjectInfo::is_inline_fast_path_eligible()` 根据 `version_id` 选择阈值
- **原因**: 与 MinIO 设计对齐，PUT 和 GET 使用相同规则

### 3. `get_stage_metrics_enabled()` 优化评估

- **结论**: 已经最优，无需改动
- **实现**: `#[inline(always)]` + `AtomicBool::load(Relaxed)`
- **开销**: 单次调用 ~1ns，可忽略

## 分析

### 1. starshard 缓存效果

小文件（1KiB-4KiB）有微小提升（+0.6% ~ +1.2%），但中大文件有退步（-4.7% ~ -7.8%）。

**可能原因**：
- starshard 的 `get()` 需要先获取 shard 锁，然后比较 `Instant::elapsed()` — 比 moka 的内置 TTL 多了一步时间计算
- starshard 的 shard 数量（64）可能不够，导致热点 bucket 的锁争用
- 对于少量 bucket 的场景（warp 只创建 1 个 bucket），moka 的简单 HashMap 可能更快

**结论**：starshard 在当前场景下未带来明显优势，可能需要调整 shard 数量或回退到 moka。

### 2. Inline 版本化规则效果

warp 测试使用非版本化 bucket，因此版本化规则对本次测试无影响。规则变更主要影响版本化场景。

### 3. 性能退步原因

10KiB-1MiB 的退步可能与：
1. **系统状态差异**：不同时间段的系统负载
2. **starshard 开销**：手动 TTL 检查比 moka 内置 TTL 略慢
3. **测试方差**：10KiB 三轮数据方差较大（22.84-24.43）

## 建议

1. **starshard 评估**：在高并发多 bucket 场景下重新评估 starshard 的优势
2. **回退选项**：如果 starshard 未带来明显优势，可回退到 moka（更简单、更快）
3. **版本化测试**：使用版本化 bucket 测试 inline 版本化规则的效果
4. **Profiling**：使用 flamegraph 定位真正的性能瓶颈
