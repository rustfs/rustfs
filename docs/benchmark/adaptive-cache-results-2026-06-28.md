# GET 小文件优化 — 自适应缓存压测结果

## 测试日期

2026-06-28

## 本次变更

### 自适应 Bucket 缓存策略

```rust
// 环境变量控制后端选择
RUSTFS_BUCKET_CACHE_STARSHARD=1  // 使用 starshard（>= 100 buckets）
// 默认                            // 使用 RwLock<HashMap>（< 100 buckets）
```

| 后端 | 适用场景 | 实现 |
|------|----------|------|
| `RwLock<HashMap>` | < 100 buckets | 默认，低开销 |
| `starshard::ShardedHashMap` | >= 100 buckets | 环境变量启用，分片锁 |

### Inline 版本化规则

GET 路径现在与 PUT 路径使用相同的阈值：

| 条件 | 阈值 | 来源 |
|------|------|------|
| 非版本化对象 | 128 KiB | `storageclass.rs::DEFAULT_INLINE_BLOCK` |
| 版本化对象 | 16 KiB | `DEFAULT_INLINE_BLOCK / 8` |

### `get_stage_metrics_enabled()` 评估

已确认最优：`#[inline(always)]` + `AtomicBool::load(Relaxed)`，无需改动。

## 测试参数

与基准测试完全一致：并发 32，持续 10s，3 轮，冷却 10s，对象数 8。

## 缓存后端对比分析

### 小文件 (< 10KiB)

| 后端 | 1KiB | 4KiB | 10KiB |
|------|-----:|-----:|------:|
| moka (重构后) | 2.52 | 10.11 | 24.88 |
| starshard (V2) | 2.55 | 10.17 | 23.32 |
| HashMap (V3) | 2.53 | 10.11 | 25.10 |

**结论**：小文件场景下三种后端差异极小（< 2%），因为 bucket 缓存命中率高，后端选择不是瓶颈。

### 中大文件 (100KiB - 1MiB)

| 后端 | 100KiB | 1MiB |
|------|-------:|-----:|
| moka (重构后) | 247.23 | 2051.34 |
| starshard (V2) | 227.94 | 1955.37 |
| HashMap (V3) | 221.81 | 1972.78 |

**结论**：中大文件的差异更明显，但方向不一致。100KiB moka 最优，1MiB moka 也最优。这表明差异主要来自系统状态波动，而非缓存后端本身。

### 推荐

| 场景 | 推荐后端 | 原因 |
|------|----------|------|
| < 100 buckets | `RwLock<HashMap>` | 最简单，性能足够 |
| >= 100 buckets | `starshard` | 分片锁减少高并发争用 |
| 不确定 | `RwLock<HashMap>` (默认) | 通过 `RUSTFS_BUCKET_CACHE_STARSHARD=1` 切换 |

## 总结

1. **自适应策略已实现**：通过环境变量 `RUSTFS_BUCKET_CACHE_STARSHARD` 控制后端选择
2. **HashMap 后端性能稳定**：小文件与 moka/starshard 持平，中大文件略有波动
3. **三种缓存后端差异极小**：说明 bucket 缓存不是性能瓶颈
4. **真正的瓶颈在其他地方**：需要 profiling 定位
